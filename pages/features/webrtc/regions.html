Regional Server Clusters

FasaFas runs SFU clusters in multiple global regions. Participants automatically connect to the nearest cluster, minimizing latency and ensuring regional data compliance.

Traffic Routing Logic
  • When you join a meeting, the signal-service checks your IP geolocation and resolves the nearest SFU cluster
  • All participants in the same meeting don't need to be in the same region — the SFU clusters communicate via Pipe Transports
  • A meeting with participants in NYC and Tokyo: NYC users → US East SFU, Tokyo users → APAC East SFU, Pipe Transport connects the two
  • This minimizes last-mile latency for each participant while keeping the meeting unified
  • STUN and TURN servers deployed in each region too — no cross-region relay for connectivity checks
  • Automatic failover: if the nearest SFU region is unavailable (>3 health check failures), the next-nearest region takes over
Regional Data Compliance
  • EU accounts: media routed exclusively through EU regions (GDPR compliance)
  • US government accounts: US regions only (FedRAMP compliance)
  • APAC accounts: APAC regions by default (data sovereignty)
  • Admin selects data residency region in Account Settings → Advanced
  • Media is not stored in transit — SFU only routes packets, does not persist them
  • Recordings stored in the account's selected S3 region (separate from SFU region)
  • Transcription AI processing region follows the account's data residency setting
