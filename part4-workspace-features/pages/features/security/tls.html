TLS 1.3 + DTLS-SRTP

All FasaFas traffic is encrypted in transit using TLS 1.3 for HTTPS APIs and DTLS-SRTP for WebRTC media streams — with perfect forward secrecy, HSTS, and certificate pinning on mobile.

🔐 TLS Configuration
Protocol versionTLS 1.3 (minimum). TLS 1.2 supported as fallback for legacy clients.
Cipher suitesTLS_AES_256_GCM_SHA384, TLS_CHACHA20_POLY1305_SHA256 — no RC4, DES, or 3DES
Key exchangeX25519 elliptic curve — fast and secure
Perfect Forward SecrecyYes — each session uses a unique ephemeral key. Past sessions can't be decrypted if a private key is compromised.
HTTPS endpointsAll API, web app, WebSocket, and webhook endpoints
Downgrade protectionHSTS header + TLS_FALLBACK_SCSV prevents protocol downgrade attacks
DTLS-SRTP for WebRTC Media
  • WebRTC audio and video streams use DTLS-SRTP — a combined handshake and encryption protocol
  • DTLS (Datagram TLS): establishes encryption keys over UDP (used because video/audio is UDP-based)
  • SRTP (Secure Real-time Transport Protocol): encrypts audio/video packets using AES-128-CM-HMAC-SHA1-80
  • Key negotiation happens before the first media packet is exchanged
  • Each participant's media stream is encrypted independently — the SFU cannot decrypt media (SRTP only — not E2EE unless E2EE mode is enabled)
  • DTLS fingerprint verification: prevents MITM attacks on WebRTC connections
Certificates & HSTS
  • Certificate: 2048-bit RSA or ECDSA P-256 issued by Let's Encrypt / DigiCert
  • Automated renewal: certificates auto-renewed 30 days before expiry — no manual intervention
  • Certificate Transparency: all certificates submitted to CT logs for public auditability
  • HSTS (HTTP Strict Transport Security): max-age=31536000; includeSubDomains; preload
  • HSTS preload: FasaFas domains submitted to browser HSTS preload lists — HTTPS enforced before first request
  • Mobile (iOS/Android): certificate pinning — app rejects certificates not matching the pinned fingerprint
