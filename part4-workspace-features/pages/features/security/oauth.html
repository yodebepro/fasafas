OAuth 2.0 PKCE

FasaFas implements OAuth 2.0 with PKCE (Proof Key for Code Exchange) for Google, Apple, and Microsoft sign-in — preventing authorization code interception without storing a client secret in the browser.

Security Considerations
OAuth 2.0 PKCE Flow
  1. User clicks 'Sign in with Google'
  2. Client generates a random code_verifier (43–128 chars) and computes code_challenge = SHA256(code_verifier)
  3. Auth request sent to Google with code_challenge — but NOT code_verifier (stays client-side)
  4. Google authenticates the user and returns an authorization code
  5. Client sends: authorization code + code_verifier to FasaFas token endpoint
  6. FasaFas verifies: SHA256(code_verifier) === code_challenge (originally registered) → tokens issued
    FasaFas calls Google's token endpoint to exchange code for identity info. Google tokens never exposed to the user's browser.

Security Properties
  • PKCE prevents authorization code interception attacks — even if the code is stolen, it can't be exchanged without the code_verifier
  • No client_secret stored in browser JS — eliminates a major attack surface for SPAs
  • State parameter: random nonce prevents CSRF attacks on the OAuth callback
  • Nonce in ID token: prevents replay of identity tokens
  • Token binding: access tokens bound to the originating device (Enterprise option)
  • Audience validation: FasaFas validates the 'aud' claim on every Google/Microsoft token
