JWT + Rotating Refresh Tokens

FasaFas uses short-lived JWT access tokens (15-minute expiry) paired with rotating refresh tokens — a compromised access token expires quickly, and refresh token rotation prevents replay attacks.

🎟 Token Design
Access tokenJWT, signed RS256, 15-minute expiry — sent in Authorization: Bearer header
Refresh tokenOpaque, 256-bit random, 30-day expiry — stored httpOnly cookie, never in localStorage
JWT payloadsub (user ID), iat, exp, jti (unique token ID for revocation), scopes, account_id
Signing keyRSA 2048-bit private key, rotated annually — public key exposed at /.well-known/jwks.json
Token validationEvery API request: verify signature, check expiry, check jti against revocation list
Clock skew tolerance60 seconds — accommodates minor clock drift between client and server
Rotating Refresh Tokens
  • When an access token expires: client sends refresh token → server issues new access token + new refresh token
  • The old refresh token is immediately invalidated — it cannot be used again
  • If an attacker tries to use a stolen refresh token after it's been rotated: the server detects the re-use and revokes the entire session family
  • Rotation prevents replay attacks: each refresh token is single-use
  • Refresh token family: all tokens issued from the same login session are linked — compromised = entire family revoked
Token Revocation
  • Immediate revocation: logout → all tokens for that session added to a Redis-backed revocation list
  • jti (JWT ID) checked against the revocation list on every API request — <5ms overhead
  • Global logout: user revokes all sessions → all tokens across all devices invalidated simultaneously
  • Admin revocation: admin can revoke all sessions for any user from Admin Panel → Users → [name] → Revoke All Sessions
  • Token lifetime is short (15 min) intentionally — even unrevoked tokens expire quickly
  • Revocation list pruned daily: expired tokens cleaned from the list automatically
