ICE Negotiation

ICE (Interactive Connectivity Establishment) is the protocol that finds and validates the best network path between your device and the SFU — testing multiple routes and picking the fastest.

How ICE Works
  1. Gather candidates
    ICE agent collects all possible IP:port pairs: local network addresses (host candidates), public IP via STUN (server reflexive), TURN relay address (relay candidate).
  2. Exchange candidates via signaling
    Each candidate is sent to the remote peer (SFU) via the signal:ice-candidate WebSocket event as it's discovered (trickle ICE).
  3. Connectivity checks (STUN binding requests)
    ICE sends STUN binding requests to every candidate pair (local × remote) in parallel. Checks with successful responses become valid pairs.
  4. Select the best pair
    Valid pairs are sorted by priority: host > server-reflexive > TURN UDP > TURN TCP. The highest-priority valid pair is used for media.
Trickle ICE — Faster Connection Setup
Why Trickle ICE matters
Traditional ICE waited until ALL candidates were gathered before starting connectivity checks — this could take 1–2 seconds. Trickle ICE sends candidates to the remote peer as they're discovered, so connectivity checks can start immediately. This typically cuts connection setup time from 1–2s to under 300ms.
  • Host candidates (LAN IPs) are discovered instantly — connectivity checks start right away
  • STUN server reflexive candidates arrive ~50–100ms later — added to the candidate list
  • TURN relay candidates arrive ~200–500ms later — added last as fallback
  • ICE selects the best working pair as soon as it has a valid candidate — doesn't wait for all
  • Result: most connections established in under 300ms from ICE start
ICE Restart — Reconnection After Network Change
  • If your device changes network (e.g., WiFi → cellular, VPN connect/disconnect), the ICE candidates may become invalid
  • ICE Restart re-runs the entire ICE gathering and connectivity check process without tearing down the session
  • Triggered automatically when: connection times out, DTLS transport fails, network interface change detected
  • From the user perspective: a brief "reconnecting…" indicator may appear for 1–3 seconds
  • Audio may drop momentarily but video session is preserved
  • If ICE Restart fails after 3 attempts, the session falls back to TURN relay
  • If all paths fail, Socket.io reconnects and a full re-join is initiated (seamless from UI perspective)
