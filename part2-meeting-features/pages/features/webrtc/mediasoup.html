Mediasoup Architecture

Mediasoup is the open-source SFU library at FasaFas's core. Combined with a Rust media server for performance-critical paths, it handles routing for hundreds of participants per room.

Mediasoup Architecture in FasaFas
Component Stack
  • Rust media server — wraps Mediasoup for performance-critical packet handling
  • Mediasoup 3.x — core SFU library (Node.js control plane, C++ media plane)
  • Workers — one Worker process per CPU core; each Worker handles multiple Routers
  • Routers — one Router per meeting room; handles all media for that room
  • WebRtcTransports — one send + one receive transport per participant
  • Producers — each participant's audio/video track (created on send transport)
  • Consumers — subscriptions: one Consumer per Producer per subscriber
Data Flow
  • Participant A publishes video → creates Producer on their send Transport
  • signal-service notifies all other participants of the new Producer
  • Each subscriber sends signal:consume → server creates Consumer on their receive Transport
  • Mediasoup routes RTP packets from Producer directly to each Consumer
  • No decoding — just packet forwarding with simulcast layer selection
  • RTCP feedback flows back from Consumer → Producer for bitrate control
Mediasoup Key Concepts
⚙️ Glossary
WorkerOS process — one per CPU core, isolated crash boundary
RouterMeeting room context — manages all Transports, Producers, Consumers in a room
WebRtcTransportICE + DTLS connection for one participant (send or receive direction)
ProducerAn outgoing media track (participant's camera or mic). Has RTP parameters.
ConsumerA subscription to a Producer — one per (subscriber, Producer) pair
PipeTransportConnect two Routers (on different servers) to bridge large meetings across machines
PlainTransportNon-WebRTC transport for recording — SFU writes raw RTP to FFmpeg
Scaling Mediasoup
  • Single Worker handles ~50–80 simultaneous participants (depends on bitrate and CPU)
  • Multiple Workers per machine — one per CPU core (a 32-core server = 32 Workers)
  • Load balancer assigns new rooms to the least-loaded Worker
  • Rooms >200 participants: PipeTransports connect Routers across multiple servers
  • Kubernetes HPA auto-scales SFU pods when CPU >70% for >60 seconds
  • Node affinity: SFU pods scheduled on bare-metal nodes (not burstable) for predictable latency
