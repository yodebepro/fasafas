What STUN Does and Why It's Needed
The NAT Problem
Most home and office devices sit behind a NAT router. The device has a private IP (e.g., 192.168.1.x) but the internet sees a different public IP (e.g., 203.0.113.42). WebRTC needs both devices to know each other's public IP + port to connect directly. STUN is how a device discovers what its public-facing IP and port look like from the internet's perspective.
  • STUN = Session Traversal Utilities for NAT (RFC 5389)
  • Your device sends a UDP packet to the STUN server
  • STUN server replies with: "I see you as 203.0.113.42:54321"
  • Your device now knows its public IP:port (called a server-reflexive candidate)
  • This candidate is shared with the SFU via ICE signaling
  • SFU tries to send media to that IP:port — if it works, the connection is established
  • No media ever flows through STUN — it's purely discovery
STUN Binding Request Flow
  1. ICE agent sends STUN Binding Request to stun.fasafas.app:3478 (UDP)
  2. STUN server reads the packet's source IP:port as seen by the internet
  3. STUN server sends a Binding Response back with the mapped address (public IP:port)
  4. ICE agent adds this as a "server reflexive" ICE candidate
  5. Candidate is sent to SFU via signal:ice-candidate WebSocket event
  6. SFU tries connecting to this IP:port — if NAT allows it, connection succeeds
