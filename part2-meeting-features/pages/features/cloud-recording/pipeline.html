Processing Pipeline

After your meeting ends, FasaFas runs a multi-stage automated pipeline to mix the recording, generate the transcript, and produce the AI summary — all without any action from you.

Complete Processing Pipeline
  1. SFU streams raw RTP tracks to S3 during meeting
    As the meeting runs, each participant's audio and video RTP streams are written as segmented files to S3 in real time. Nothing to process when meeting ends — tracks already in cloud.
  2. meeting.ended Kafka event triggers pipeline
    When host clicks End Meeting, the meeting-service publishes a meeting.ended event to Kafka. The media-service's Lambda consumer picks it up and starts the pipeline.
  3. ECS Fargate task launched for FFmpeg mixing
    A Fargate task with the FFmpeg container is spun up. It reads all participant track segments from S3, mixes them according to the active-speaker timeline, and produces: MP4 (video+audio), M4A (audio only), and chat TXT.
  4. HLS segments generated in parallel
    While the MP4 is being mixed, an HLS (.m3u8 playlist with .ts chunks) is generated from the raw tracks so you can start watching via streaming before the MP4 download is ready.
  5. recording.completed Kafka event fired
    Files uploaded to recordings S3 bucket. recording.completed event published to Kafka. ai-companion-service and notification-service consume this event.
  6. Whisper transcription job (AWS Batch)
    Audio extracted from MP4 → sent to Whisper large-v3. Speaker diarization runs in parallel (Pyannote.audio). Both results merged into speaker-labeled VTT and JSON transcript files.
  7. Claude AI: summary + chapters + action items
    Full labeled transcript sent to Claude Sonnet. Returns: 3–5 paragraph summary, 5–10 smart chapters with titles, action items list, key decisions. All written to Postgres + S3.
  8. Host notification + webhook fired
    meeting.summary.completed webhook sent to customer endpoint. WebSocket push to host's active browser: "Your meeting summary is ready." Hub updates with all assets.
Expected Processing Times
⏱️ Approximate Pipeline Timing
HLS stream ready~1–2 minutes after meeting end (watch while processing)
MP4 download ready~2–5 minutes (depends on meeting duration; 1 hr meeting ≈ 3 min mix)
Transcript ready~5–8 minutes (Whisper processes at ~0.5× real-time speed)
AI summary ready~8–15 minutes (Claude processes full transcript in ~30–90 seconds)
Total pipelineTypically 10–20 minutes for a 1-hour meeting; up to 40 min for 3+ hours
Failure retryEach stage retries up to 3 times with exponential backoff before alerting
Pipeline Error Handling
  • Each pipeline stage is independently idempotent — can retry any stage without corrupting output
  • If FFmpeg mix fails: alert sent to infrastructure team; host notified to retry from Hub
  • If Whisper fails: transcript marked as "Processing failed" — host can manually request re-run
  • If Claude fails: summary generation retried up to 3 times; if all fail, summary shows "Generation failed — try regenerate"
  • Partial success: if transcript succeeds but summary fails, transcript is still available
  • Long meetings (>4 hours): Whisper job split into 30-minute chunks and run in parallel
  • Status updates visible in Hub → Recordings → processing indicator per recording
